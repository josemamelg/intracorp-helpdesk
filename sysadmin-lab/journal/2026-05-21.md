## Objetivo

Levantar la aplicacion con Docker Compose, validar los servicios principales y entender como diagnosticar una falla basica detras de Nginx.

## Servicios revisados

- `nginx`: reverse proxy publico en `localhost:8080`.
- `frontend`: interfaz React servida internamente.
- `backend`: API Node.js/Express.
- `postgres`: base de datos PostgreSQL.


## Health checks aprendidos

```text
/health        -> valida Nginx
/api/health    -> valida Nginx + backend
/api/health/db -> valida Nginx + backend + PostgreSQL
```

## Incidente simulado

Se detuvo el backend para observar como responde el sistema:

```bash
docker compose stop backend
```

Al probar:

```bash
curl http://localhost:8080/health
curl http://localhost:8080/api/health
curl http://localhost:8080/api/health/db
```

se observo:

- `/health` respondio `nginx ok`.
- `/api/health` fallo con `504 Gateway Time-out`.
- `/api/health/db` fallo con `504 Gateway Time-out`.

## Diagnostico

Nginx seguia funcionando, pero las rutas `/api` no respondian porque el backend estaba detenido o inaccesible.

Conclusion:

```text
Nginx operativo.
Backend no disponible.
PostgreSQL no se puede validar desde la app mientras el backend esta caido.
```

## Recuperacion

```bash
docker compose start backend
curl http://localhost:8080/api/health
curl http://localhost:8080/api/health/db
```

Despues de iniciar el backend, la API y la conexion a PostgreSQL volvieron a responder correctamente.

## Aprendizajes

- Una ruta HTTP no es lo mismo que un comando de Linux.
- `curl` permite probar endpoints desde la terminal.
- Un reverse proxy puede estar vivo aunque el backend este caido.
- Un error `504 Gateway Time-out` indica que Nginx no recibio respuesta del upstream a tiempo.
- Los health checks ayudan a ubicar en que capa esta el problema.
