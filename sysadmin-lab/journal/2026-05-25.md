# 2026-05-25 - Dia 4

## Practica

Backup y restore real de PostgreSQL.

## Objetivo

Validar que un backup no solo se genera, sino que tambien puede restaurarse correctamente.

## Comandos usados

```bash
./scripts/check-health.sh
./scripts/backup-postgres.sh
ls -lh backups/
docker compose exec postgres psql -U intracorp intracorp_helpdesk
./scripts/restore-postgres.sh backups/intracorp-helpdesk-20260525-071351.sql.gz
```

## Prueba realizada

Se genero un backup y luego se modifico el ticket `#1`:

```sql
UPDATE tickets SET status = 'closed' WHERE id = 1;
SELECT id, title, status FROM tickets WHERE id = 1;
```

El ticket quedo temporalmente como:

```text
closed
```

Despues se intento restaurar el backup.

## Hallazgo

El primer restore produjo errores porque el dump SQL completo se intento cargar encima de una base que ya tenia tablas, tipos, indices y datos.

Errores observados:

```text
type already exists
relation already exists
duplicate key value violates unique constraint
multiple primary keys are not allowed
```

Aunque el script terminaba mostrando `Restore aplicado`, el dato no habia vuelto al estado del backup.

El ticket `#1` seguia:

```text
closed
```

## Correccion aplicada

Se actualizo el script de restore para limpiar el schema `public` antes de cargar el dump:

```sql
DROP SCHEMA IF EXISTS public CASCADE;
CREATE SCHEMA public;
```

Luego se agrego `ON_ERROR_STOP=1` para que `psql` falle si aparece un error durante la restauracion.

## Validacion final

Se inspecciono el backup y se confirmo que el ticket `#1` estaba guardado como:

```text
in_progress
```

Despues del restore corregido:

```sql
SELECT id, title, status FROM tickets WHERE id = 1;
```

Resultado:

```text
1 | VPN no conecta desde fuera de oficina | in_progress
```

Tambien se validaron health checks:

```bash
./scripts/check-health.sh
```

Resultado:

```text
All health checks passed.
```

## Aprendizajes

- Un backup debe probarse con restore, no solo generarse.
- Un mensaje final del script no garantiza que todo salio bien si no se controla el codigo de error.
- Restaurar un dump completo sobre una DB existente puede generar duplicados.
- En un laboratorio es valido limpiar el schema antes de restaurar, pero en produccion se debe planificar con mas cuidado.
- `ON_ERROR_STOP=1` ayuda a evitar falsos positivos en scripts con `psql`.

## Pendiente

- Crear una practica de restore en una base temporal para no pisar la base activa.
- Agregar un script de verificacion post-restore.

# 2026-05-25 - Monitoreo con Uptime Kuma

## Practica

Agregar Uptime Kuma al stack Docker y definir monitores HTTP para el laboratorio IntraCorp Helpdesk.

## Objetivo

Monitorear la app desde la misma red Docker donde corren `nginx`, `backend` y `postgres`, evitando confundir URLs publicadas hacia el host con URLs internas entre contenedores.

## Servicio agregado

Se agrego el servicio `uptime-kuma` a `docker-compose.yml`:

```text
Imagen: louislam/uptime-kuma:1
Puerto host: 3001
Puerto contenedor: 3001
Volumen: uptime_kuma_data:/app/data
Red: intracorp_net
```

Acceso desde el host:

```text
http://localhost:3001
```

## Monitores definidos

Como Uptime Kuma corre dentro de Docker, las URLs correctas son nombres DNS internos de Compose:

```text
http://nginx/health
http://backend:4000/api/health
http://backend:4000/api/health/db
http://nginx/
```

Monitores recomendados:

| Nombre | URL |
| --- | --- |
| IntraCorp - Nginx health | `http://nginx/health` |
| IntraCorp - API health | `http://backend:4000/api/health` |
| IntraCorp - DB health | `http://backend:4000/api/health/db` |
| IntraCorp - Web app | `http://nginx/` |

## Validacion interna

Se valido la conectividad desde un contenedor de la red Docker:

```bash
docker compose exec nginx sh -lc 'wget -qO- http://nginx/health'
docker compose exec nginx sh -lc 'wget -qO- http://backend:4000/api/health'
docker compose exec nginx sh -lc 'wget -qO- http://backend:4000/api/health/db'
docker compose exec nginx sh -lc 'wget -q --spider http://nginx/'
```

Resultados observados:

```text
nginx ok
{"status":"ok","service":"intracorp-helpdesk-api",...}
{"status":"ok","database":"reachable",...}
```

El monitor de `http://nginx/` se valida por codigo de salida `0`.

## Aprendizajes

- `localhost` dentro de Uptime Kuma apunta al contenedor de Kuma, no al host Windows ni a Nginx.
- Para monitoreo entre contenedores conviene usar DNS interno de Docker Compose.
- `http://localhost:3001` sirve para entrar a Kuma desde el host, pero no debe usarse como URL de los monitores internos.
- Separar monitores de proxy, API y DB ayuda a ubicar rapidamente donde esta la falla.
