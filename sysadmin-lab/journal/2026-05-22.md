# 2026-05-22 - Dia 2

## Practica

Simulacion de caida de PostgreSQL dentro del laboratorio IntraCorp Helpdesk.

## Objetivo

Entender que ocurre cuando la base de datos deja de estar disponible y como se refleja el problema en health checks, logs y comportamiento de la aplicacion.

## Comandos usados

```bash
docker compose ps
curl http://localhost:8080/health
curl http://localhost:8080/api/health
curl http://localhost:8080/api/health/db
docker compose stop postgres
docker compose logs --tail=40 backend
docker compose logs --tail=40 postgres
docker compose start postgres
```

## Health checks revisados

```text
/health        -> valida Nginx
/api/health    -> valida Nginx + backend
/api/health/db -> valida Nginx + backend + PostgreSQL
```

## Observaciones

Al detener PostgreSQL, Nginx puede seguir respondiendo su propio endpoint `/health`.

El backend puede seguir vivo como proceso, pero las funciones que necesitan base de datos dejan de operar correctamente.

El endpoint mas util para detectar el problema es:

```bash
curl http://localhost:8080/api/health/db
```

porque valida el camino completo:

```text
Nginx -> Backend -> PostgreSQL
```

## Impacto esperado

Con PostgreSQL caido pueden fallar:

- Login.
- Dashboard.
- Tickets.
- Gestion de usuarios.
- Auditoria.
- Health check de base de datos.

La interfaz o Nginx pueden seguir cargando, pero la aplicacion queda parcialmente inutilizable porque los datos dependen de PostgreSQL.

## Diagnostico

Si `/health` responde pero `/api/health/db` falla, el problema no esta en Nginx.

Si `/api/health` responde pero `/api/health/db` falla, el backend esta vivo pero no puede usar la base de datos.

Conclusion:

```text
Nginx operativo.
Backend operativo o parcialmente operativo.
PostgreSQL detenido o inaccesible.
```

## Errores esperados en logs

En logs del backend se pueden observar errores relacionados con conexion a PostgreSQL, por ejemplo:

```text
ECONNREFUSED
connection refused
SQL query failed
timeout
database system is shutting down
```

## Recuperacion

```bash
docker compose start postgres
docker compose ps
curl http://localhost:8080/api/health/db
```

La recuperacion se valida cuando `/api/health/db` vuelve a responder con:

```json
{
  "status": "ok",
  "database": "reachable"
}
```

## Aprendizajes

- Un servicio puede estar vivo aunque una dependencia critica este caida.
- No alcanza con validar Nginx o backend; tambien hay que validar dependencias.
- `/api/health/db` es el endpoint mas importante para confirmar estado real de la base.
- Una caida de PostgreSQL afecta funciones de negocio aunque la web siga cargando.
- Los logs del backend son clave para confirmar errores de conexion a DB.
